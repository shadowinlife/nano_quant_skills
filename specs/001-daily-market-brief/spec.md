# Feature Specification: A股开盘核心新闻聚合研究 Skill

**Feature Branch**: `001-add-core-news-skill`  
**Created**: 2026-04-28  
**Status**: Draft  
**Input**: User description: "核心: 我想要创建一个新的SKILLS, 它的任务是执行每日最核心新闻的聚合分析, 为A股市场开盘投资提供基本的精炼内容. 它的分析路径如下: 1. 如果美股昨日交易, 采集美股昨日的操作热点 2. 采集财经媒体昨日的主线信息, 找到交叉高光的不多于5个信息 3. 基于指定配置表, 采集自媒体评论要点并总结过去5个交易日共同关注点 4. 基于指定配置表, 采集昨日新发布的研报, 覆盖海外和国内投研机构 5. 基于指定配置表, 追踪大宗商品价格与主要产区新闻 6. 汇总各子任务中间文件形成最终独立报告；同时对任务进行 step by step 规划、拆解自动执行与人工介入部分，并逐步探索稳定数据源。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 生成开盘前精炼简报 (Priority: P1)

作为 A 股盘前研究者，我需要在开盘前快速获得一份独立的核心新闻聚合报告，覆盖隔夜海外热点、主流财经媒体交叉主线、自媒体共同关注点、最新研报和大宗商品变化，以便在有限时间内完成当日重点跟踪与初步交易准备。

**Why this priority**: 这是该 Skill 的核心交付物；如果不能稳定产出一份可读、可筛选的盘前简报，其他配置能力和自动化迭代都没有独立价值。

**Independent Test**: 仅执行一次完整日报流程，即可验证系统是否能基于前一交易日数据产出结构完整、重点明确、可直接阅读的盘前报告，并明确每部分内容来源与置信度。

**Acceptance Scenarios**:

1. **Given** 前一交易日存在可用的海外市场、财经媒体、研报和大宗商品信息，**When** 研究者运行当日简报流程，**Then** 系统生成一份独立报告，包含所有已启用分析模块的摘要、优先级排序和缺失说明。
2. **Given** 美股前一日未交易，**When** 研究者运行当日简报流程，**Then** 报告明确标注该模块跳过原因，并仍输出其他模块的完整结果。
3. **Given** 某一来源当日不可用，**When** 系统汇总结果，**Then** 报告保留已完成模块内容，并在对应部分标注数据缺口、替代说明或待人工补充状态。

---

### User Story 2 - 逐步沉淀自动化研究流程 (Priority: P2)

作为维护该 Skill 的研究工程人员，我需要把日报任务拆解成可执行子任务，并为每个子任务区分自动执行部分与人工介入部分，逐步收敛为可复用、可维护的自动研究流程。

**Why this priority**: 用户明确要求先形成 step by step 规划，再迭代到自动研究形态。没有分阶段拆解，就无法判断哪些部分应优先自动化、哪些部分仍需人工质检。

**Independent Test**: 查看 Skill 输出的规划说明，确认每个子任务都具有明确目标、输入、输出、自动化边界、人工审核点和下一阶段演进方向。

**Acceptance Scenarios**:

1. **Given** 新建该 Skill 的初始版本，**When** 用户查看任务规划，**Then** 每个分析模块都附带阶段目标、可执行子任务列表和自动/人工分工说明。
2. **Given** 某个模块当前尚不能完全自动化，**When** 用户查看规划，**Then** 文档明确指出当前人工介入环节、触发条件和未来自动化改进方向。

---

### User Story 3 - 基于配置稳定扩展信息覆盖面 (Priority: P3)

作为研究流程维护者，我需要通过配置表管理需要跟踪的自媒体、研报机构和大宗商品主题，使研究范围能够按主题扩展，而不需要每次重写流程。

**Why this priority**: 用户要求多个分析模块依赖指定配置表。只有配置可扩展，Skill 才能持续适配新的观察名单和跟踪主题。

**Independent Test**: 修改配置表中的来源或跟踪对象后，重新执行流程并确认报告内容范围发生对应变化，同时保留统一的输出结构。

**Acceptance Scenarios**:

1. **Given** 用户新增一个自媒体观察对象到配置表，**When** 下一次运行流程，**Then** 系统在自媒体分析部分纳入该对象并说明其是否参与共同关注点判断。
2. **Given** 用户调整研报机构或大宗商品观察清单，**When** 重新执行流程，**Then** 输出仅覆盖配置中启用的对象，并保留未命中对象的状态说明。

### Edge Cases

- 当某一模块的可用来源少于预期时，系统如何保证仍能输出可读摘要，而不是整份报告失败？
- 当多来源信息相互矛盾时，系统如何标注分歧、避免把未证实观点当作主线结论？
- 当自媒体、研报或大宗商品配置表为空、过期或部分失效时，系统如何提示并限制错误传播？
- 当同一主题在不同模块中重复出现时，系统如何合并重复表述并保留最重要证据？
- 当报告生成时间临近开盘、部分子任务超时或人工审核未完成时，系统如何产出可用的阶段性结果？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个按日执行的研究流程，用于生成面向 A 股开盘前决策的核心新闻聚合报告。
- **FR-002**: 系统 MUST 在适用情况下分析前一交易日海外市场的热点主题，并在该市场未交易时明确跳过原因。
- **FR-003**: 系统 MUST 汇总多个主流财经媒体前一日的主线信息，并提炼出不多于 5 个交叉高光主题。
- **FR-004**: 系统 MUST 基于用户维护的配置表跟踪指定自媒体来源，并识别这些来源在过去 5 个交易日内重复关注的共同主题。
- **FR-005**: 系统 MUST 基于用户维护的配置表追踪前一日新发布的研报，且覆盖境内外研究机构。
- **FR-006**: 系统 MUST 基于用户维护的配置表追踪重点大宗商品的价格变化及主要产区相关新闻，并总结对当日市场观察的潜在影响。
- **FR-007**: 系统 MUST 为每个分析模块生成结构化中间结果，以支持独立审阅、错误定位和最终汇总。
- **FR-008**: 系统 MUST 将所有中间结果汇总为一份独立报告，报告中需区分确认信息、待核实信息和缺失信息。
- **FR-009**: 系统 MUST 为该 Skill 提供 step by step 的执行规划，将整体流程拆解为多个核心子任务，并为每个子任务定义输入、输出、完成标准和依赖关系。
- **FR-010**: 系统 MUST 对每个子任务明确区分自动执行部分与人工介入部分，并说明人工介入的触发条件、审核目标和退出条件。
- **FR-011**: 系统 MUST 支持按阶段迭代自动化能力，使用户能够从半自动研究流程逐步过渡到更完整的自动研究流程。
- **FR-012**: 系统 MUST 为数据来源建立稳定性探索路径，能够记录候选来源类别、适用范围、可持续性风险和替代方案。
- **FR-013**: 系统 MUST 允许用户通过配置变更来扩展或收缩跟踪范围，而不改变最终报告的总体结构。
- **FR-014**: 系统 MUST 采用分阶段发布策略，在关键模块结果已生成时允许先产出临时版报告，并在后续人工复核或迟到模块完成后补充修订版。
- **FR-015**: 系统 MUST 将首版数据来源分为“生产源”和“探索源”两类，默认仅依赖稳定接口或标准化订阅源生成正式内容，定制抓取方案仅作为未启用的候选补充路径进行记录与评估。
- **FR-016**: 系统 MUST 为首版设定最小可用覆盖范围，按模块维护一组核心跟踪清单，优先保证核心清单的稳定交付，再逐步扩展到更广范围。
- **FR-017 (Portability)**: Skill/MCP/script behavior MUST remain compatible across macOS, Linux, and Windows, or MUST document explicit fallback limits.
- **FR-018 (Local Setup)**: Feature MUST provide a lightweight local setup flow suitable for open-source users with minimal configuration steps.
- **FR-019 (Opencode Validation)**: Feature MUST define local opencode validation commands and objective pass criteria before commit.
- **FR-020 (Commit Hygiene)**: Feature deliverables and workflows MUST prevent IDE metadata, temporary files, local absolute paths, and sensitive local configuration from entering repository history.

### Key Entities *(include if feature involves data)*

- **研究日报任务**: 一次面向特定交易日执行的完整聚合分析，包含执行时间、覆盖模块、完成状态和最终报告。
- **分析模块结果**: 某一子任务生成的中间结果，包含摘要、证据来源、时间范围、状态标记和人工复核需求。
- **跟踪配置表**: 用户维护的观察清单，定义需要跟踪的自媒体、研报机构、大宗商品主题及启用状态。
- **高光主题**: 在多个来源中被重复提及、适合纳入盘前报告重点关注的市场主题。
- **来源稳定性记录**: 针对候选数据来源维护的评估信息，包含覆盖范围、可靠性、成本、故障模式和替代策略。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 研究者在单次执行后可于 15 分钟内阅读完整日报，并定位当日最重要的 5 个以内主题与其来源依据。
- **SC-002**: 在连续 20 个交易日内，日报流程至少 90% 的交易日能够产出包含已启用模块结果的独立报告，即使个别来源缺失也不阻断整体交付。
- **SC-003**: 对于配置表中已启用的跟踪对象，至少 95% 能在对应模块结果中得到“已覆盖、无新增或来源缺失”三类之一的明确状态。
- **SC-004**: 维护者能够在不改动流程主体的前提下，于 10 分钟内完成一次跟踪清单调整，并在下一次执行结果中看到范围变化。
- **SC-005**: 每个子任务都具备可审阅的自动执行说明和人工介入说明，使新维护者能够在 30 分钟内理解全流程的阶段划分与责任边界。

## Assumptions

- 该 Skill 的首要目标是生成盘前研究材料，而非直接输出交易指令或投资建议。
- 初始版本允许部分模块保留人工复核或人工补录环节，但必须显式标注在规划与报告中。
- 用户能够维护必要的跟踪配置表，并接受不同模块在早期阶段采用不同成熟度的数据来源。
- 海外市场热点、主流媒体主线、自媒体共识、研报动态与大宗商品变化共同构成该日报的最小可用范围。
- 对数据来源的探索以“稳定可持续”为优先，允许不同来源类别并行评估，只要最终报告能清楚标明证据质量与缺口。
- 默认采用“关键模块先出临时版、后续补修订版”的发布策略，以优先满足盘前时效需求。
- 首版默认采用“生产源/探索源”分层策略，其中探索源只作为候选补充路径记录，不直接进入正式输出。
- 首版默认按模块维护核心清单作为最小可用覆盖范围，扩展清单在后续迭代中逐步纳入。
